 DOCUMENTO DE ESPECIFICACIÓN TÉCNICA Y REQUERIMIENTOS: "GESTI"
Introducción, Ecosistema y Arquitectura
1. La Historia de "Gesti"
Gesti nace como una solución para la gestión y optimización del tiempo en microempresas y
 profesionales independientes que brindan servicios basados en turnos o clases (centros de yoga/pilates, peluquerías, consultorios, etc.).
 El problema detectado es la dispersión de la información (chats de WhatsApp perdidos, cuadernos físicos y planillas de Excel difíciles de auditar). 
 Gesti centraliza la reserva, el control automático de regularidad del cliente, 
 el historial médico o de preferencias, y la analítica diaria/mensual en una Single-Page Application (SPA) intuitiva, 
 permitiendo que el administrador se enfoque en su propuesta de valor y no en la carga manual de datos.

2. Tecnologías, Bibliotecas y Arquitectura 
El desarrollo se estructurará de la siguiente manera:
Framework Principal: Angular (versión 15+) configurado como SPA (Single Page Application) para transiciones fluidas sin recarga de pantalla.
Lenguaje: TypeScript, garantizando tipado estricto para las interfaces de usuarios, turnos y categorías.
Maquetación y Estilos: HTML5 y CSS3 nativo utilizando CSS Grid y Flexbox para el dashboard adaptivo, evitando sobrecarga de frameworks visuales pesados.
Bibliotecas Clave:
@fullcalendar/angular: Para el renderizado óptimo y nativo del calendario.
chart.js / ng2-charts: Para las barras de analítica mensual y diaria.
Arquitectura Frontend: Arquitectura basada en componentes desacoplados. 
Se utilizarán Servicios de Angular (@Injectable) para centralizar la lógica de negocio y las peticiones HTTP, separándolas de la vista.

3. Consumo de API y Almacenamiento de Datos
¿Dónde se guardan los datos?: En una base de datos relacional (PostgreSQL).
¿De dónde se consume la API?: Se creará un Backend propio en Node.js (Express) que expondrá una API REST con formato JSON. El Frontend de Angular la consumirá mediante el módulo HttpClient inyectado en los servicios correspondientes.
Requerimientos Funcionales y No Funcionales por Módulo

1. Módulo Calendario (Home)
Este módulo es el centro visual donde se visualizan las reservas de turnos o clases.
Acciones y Requerimientos Funcionales:
RF1.1 - Visualización del Calendario: El sistema debe permitir visualizar las clases/turnos reservados en vistas de Mes, Semana y Día.
RF1.2 - Bloques por Categorías: Cada turno debe mostrar de forma sutil el color configurado para su categoría (ej. Verde para Pilates, Azul para Yoga).
RF1.3 - Vista de Datos Básicos: Al hacer clic o pasar el cursor por un turno, se deben desplegar los datos básicos (Hora, Profesor, Cantidad de Alumnos inscriptos).
RF1.4 - Sector de Métrica Estilo Barra: En el  módulo se expondrán en tiempo real la cantidad de personas anotadas en el mes actual y la cantidad de personas en el día seleccionado.

Requerimientos No Funcionales:
RNF1.1: El renderizado de los eventos del calendario no debe demorar más de 1.5 segundos al cambiar entre meses.

2. Módulo Perfil de Usuario
Acciones y Requerimientos Funcionales:
RF2.1 - Visualización de Datos Personales:  Nombre, Apellido y Número de contacto.
RF2.2 - Sector Notas Dinámicas: Sección donde el administrador visualiza comentarios críticos guardados (ej. "Faltó porque se quebró el pie", "Objetivo: Tonificar", "Nivel: Principiante", o "Cosas que deberíamos saber").
RF2.3 - Automatización de Respuestas Form: El sistema procesará las respuestas del Google Form de feedback enviado automáticamente post-clase e indexará directamente los comentarios en esta sección.
RF2.4 - Cálculo de Regularidad: El sistema calculará automáticamente un promedio de asistencia basado en los días anotados versus asistidos, asignando un tag visual: Regular (≥ 80%), Medio (50% - 79%) o No Regular (< 50%).

3. Módulo Historial y Notas 
Acciones y Requerimientos Funcionales:
RF3.1- Métricas de Deserción: Mostrar gráficos de barras comparativos entre la cantidad de personas que reservaron frente a las que efectivamente asistieron o faltaron.
RF3.2 - Registro de Anotaciones Internas: Espacio de bloc de notas exclusivo del Admin para dejar asentados cambios de planificación (ej. "Modificar ejercicios de la clase de piernas", "Comprar colchonetas").

4. Módulo Admin 
Acciones y Requerimientos Funcionales:
RF4.1 - Creador de Categorías: Formulario para añadir disciplinas o servicios (ej. yoga o pilates), asignándoles un nombre y un código de color.
RF4.2 - Configuración de Turnos Diarios: Permitir definir las franjas horarias manejadas por el negocio (ej. establecer 3 turnos al día: 10:00-11:00 / 15:00-16:00 / 18:00-19:00).
RF4.3 - Asignación de Recursos: Input para ingresar el nombre de la profesora/profesional a cargo del turno.
RF4.4 - Límite de alumnos : Definir una cantidad máxima de alumnos/clientes permitidos por hora para bloquear automáticamente inscripciones excedentes (ej. evitar 50 personas en una hora).

5. Módulo Perfiles 
Acciones y Requerimientos Funcionales:
RF5.1 - Agrupación por Categorías: Mostrar a los usuarios inscriptos ordenados en filas según su disciplina (ej. Fila de Pilates mostrando todas las inscriptas de esa categoría).
RF5.2 - Purga Automática de Inactividad: Mostrar inscripciones activas con un rango de 2 meses. Si un usuario registra inactividad total o faltas consecutivas por más de 3 meses, el sistema lo eliminará automáticamente del listado.


Integraciones Externas e Importaciones por Módulo
1. Estrategia de Flujo de Captación (Onboarding) y Consultas
Captación Inicial: El negocio dispondrá de un bot de recolección de info por WhatsApp o un código QR físico en el local que redireccionará a un Google Form con las preguntas clave: Nombre, Apellido, Número de teléfono y Horario del turno.
Automatización del Consumo: Mediante Google Apps Script, las respuestas registradas en la planilla de Google Sheets asociada se enviarán en formato JSON a la API de nuestra base de datos para impactar directamente en el sistema.
